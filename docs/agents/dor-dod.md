# 就绪定义（DoR）

一个任务在以下条件全部满足时，才可进入实现阶段：

1. **需求以清晰、可测试的术语陈述。** 对"完成"的样子没有歧义。
2. **验收标准已定义。** 每个标准是一个布尔检查（通过/失败）。
3. **受影响的模块/系统已识别。** 任务列出了将变更的文件或系统。
4. **依赖已解决或明确标注。** 被阻塞的任务标注了阻塞 issue。
5. **范围适合垂直切片。** 一个小型、可运行的特性 —— 不是水平分层。
6. **没有会阻塞 TDD 循环的开放问题。** 如果不确定，路由回 grill-me。

如果以上任何一项缺失，不要开始实现。路由回：
- Phase 1（grill-me）处理不清晰的需求。
- Phase 2（to-prd / to-issues）处理缺失的验收标准或过大的范围。

# 完成定义（DoD）

一个任务只有在以下条件**全部**满足时才算完成：

1. **请求的行为已实现。** 代码完成了 issue 所描述的功能。
2. **测试已添加或更新。**
   - JSON 数据文件如果被修改，已验证。
   - 场景文件（.tscn）在编辑器中可无错误打开。
   - 如果 GUT/GdUnit4 可用：编写 GDScript 单元测试。
   - 如果没有测试运行器：文档化手动验证步骤。
   - 预期行为已作为结构化注释写在代码中。
3. **质量门禁通过。** `./scripts/agent-check.sh` 以退出码 0 退出。
   如果质量门禁失败，不要声称完成。修复问题并重新运行。
4. **没有修改无关文件。** diff 只包含与任务相关的变更。
5. **文档已更新**，当行为、命令或架构发生变更时。
6. **最终响应总结：**
   - 改了什么和为什么。
   - 改了哪些文件。
   - 运行了哪些命令及其结果。
   - 有哪些已知风险或后续工作。

---

## 质量门禁替代方案

如果 `scripts/agent-check.sh` 不存在或无法运行：

1. 手动运行可用的检查（JSON 验证、手动场景测试）作为单独命令。
2. 报告哪些检查通过、哪些被跳过。
3. 在提交消息或 PR 描述中注明质量门禁不可用。
